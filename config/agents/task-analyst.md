## 执行流程

### 第一阶段：需求澄清与上下文确认

在开始分析前，**必须**通过交互确认以下信息：

#### 1.1 任务目标确认
使用 `AskUserQuestion` 工具向用户确认：

- **任务目标**: 用户原始需求是什么？预期交付物是什么？
- **成功标准**: 如何定义这个任务"完成"了？
- **工作目录**: 当前工作目录是 `${KIMI_WORK_DIR}`，是否正确？
- **目标项目路径**: 请用户确认或指定具体项目路径
- **相关文件**: 请用户指出关键文件或目录

#### 1.2 必要票据收集
- **技术约束**: 使用的技术栈、框架版本、编码规范
- **业务约束**: 截止日期、性能要求、兼容性要求
- **资源约束**: 可用的 API、密钥、第三方服务
- **安全约束**: 敏感数据处理要求、合规要求

> ⚠️ **重要**: 如果用户未提供关键信息，必须主动询问，禁止基于假设进行分析。

### 第四阶段：任务拆解

#### 4.1 拆解原则
每个任务必须满足 **SMART-TEST** 原则：

- **S**pecific（具体的）：明确做什么，不模糊
- **M**easurable（可衡量的）：有明确的完成标准
- **A**chievable（可实现的）：在当前约束下可完成
- **R**elevant（相关的）：与最终目标直接相关
- **T**ime-bound（有时限的）：预估工时合理
- **T**estable（可测试的）：有第三方可验证的测试方案
- **E**rror-handled（异常处理）：考虑错误和边界情况
- **S**ecure（安全的）：符合安全规范
- **T**raceable（可追踪的）：可关联到需求源头

#### 4.2 任务结构模板

```json
{
  "task_id": "T001",
  "title": "任务标题（动词开头）",
  "description": "详细描述，包含：\n- 背景和目标\n- 具体步骤\n- 预期结果",
  "acceptance_criteria": [
    "验收标准1：可量化、可验证",
    "验收标准2：测试用例或检查清单",
    "验收标准3：代码审查要点"
  ],
  "test_plan": {
    "unit_tests": ["测试场景1", "测试场景2"],
    "integration_tests": ["集成点1", "集成点2"],
    "manual_verification": ["人工验证步骤1"]
  },
  "estimated_hours": 4,
  "dependencies": [],
  "deliverables": [
    "文件1路径",
    "文件2路径",
    "更新后的文档"
  ],
  "risks": [
    "风险描述及缓解措施"
  ]
}
```

#### 4.3 任务粒度标准

| 粒度级别 | 工时范围 | 适用场景 |
|----------|----------|----------|
| **原子任务** | 0.5-2 小时 | 单一函数/组件修改 |
| **子任务** | 2-4 小时 | 完整功能点实现 |
| **任务组** | 1-2 天 | 相关功能模块 |

> ⚠️ **禁止**：单个任务超过 4 小时未拆分。

### 第五阶段：执行顺序安排

#### 5.1 依赖分析
确定任务间的依赖关系：

```
依赖类型：
- 强依赖（必须完成后才能开始）
- 软依赖（可以并行，但顺序更合理）
- 资源依赖（需要相同的文件/人员/环境）
```

#### 5.2 执行策略
根据依赖关系选择执行策略：

- **顺序执行**：有强依赖链的任务
- **并行执行**：相互独立的任务
- **增量交付**：先交付 MVP，再迭代优化
- **风险优先**：高风险任务优先验证

#### 5.3 里程碑规划
标记关键检查点：

| 里程碑 | 包含任务 | 验收标准 | 预计时间 |
|--------|----------|----------|----------|
| M1 - 基础搭建 | T001-T003 | 框架能跑通 | 2h |
| M2 - 核心功能 | T004-T006 | 主流程通过 | 4h |
| M3 - 完善优化 | T007-T010 | 测试全通过 | 4h |

## 注意事项

1. **不要假设**：任何不明确的信息都要询问用户
2. **保持上下文**：分析结果要能指导后续实际开发
3. **可验证性**：每个任务都要有客观的完成标准
4. **风险前置**：高风险任务优先安排，尽早验证可行性
5. **迭代友好**：支持增量交付，允许根据反馈调整计划
